# 2026.05.02.01 - Fix building XP percentage display at level cap

Status: Done

## Goal

Buildings at the current level cap incorrectly display 100% progress instead of their
actual XP percentage toward the next level.

## Bug Description

In `build_village_embed` inside `src/cogs/ui_renderer.py`:

```python
next_req = (blevel + 1) * xp_per_level
bpct = 100 if blevel >= level_cap else math.floor(bxp / next_req * 100)
```

When `blevel >= level_cap`, `bpct` is unconditionally set to `100`.

However, `building_manager.add_xp` clamps `xp_progress` to `level * xp_per` when at cap
(not to `(level + 1) * xp_per`). For a Lv1 building with `BUILDING_XP_PER_LEVEL=1000`:

- `xp_progress` is clamped to `1 * 1000 = 1000`
- `next_req = (1 + 1) * 1000 = 2000`
- Actual percentage: `floor(1000 / 2000 * 100) = 50%`

The cap override forces 100%, hiding the real progress value.

Observed (stages_cleared=3, level_cap=1, all buildings at Lv1):
```
🌾 採集場 Lv1 (100%)
🔨 加工廠 Lv1 (100%)
⚔️ 狩獵場 Lv0 (40%)
🔬 研究所 Lv1 (100%)
```

Expected (採集場 xp_progress=1000, next_req=2000):
```
🌾 採集場 Lv1 (50%)
```

## Scope

### 1. Remove the level-cap override in `build_village_embed`

File: `src/cogs/ui_renderer.py`

Current:
```python
next_req = (blevel + 1) * xp_per_level
bpct = 100 if blevel >= level_cap else math.floor(bxp / next_req * 100)
```

New:
```python
next_req = (blevel + 1) * xp_per_level
bpct = math.floor(bxp / max(next_req, 1) * 100)
```

- Remove the `100 if blevel >= level_cap else` condition entirely.
- Guard against division-by-zero with `max(next_req, 1)` (consistent with the stage
  progress formula on line 91).
- The `level_cap` local variable is still needed for the `公用設施 (等級上限：Lv{level_cap})`
  header line; do not remove it.

### 2. Update the affected test

File: `tests/test_discord_commands.py`, `test_building_rows_are_plain_percentage_only`

The test passes buildings with `xp_progress=50` (gathering_field) against
`BUILDING_XP_PER_LEVEL=1000` and `stages_cleared=3` (level_cap=1). After the fix:

- `next_req = 2 * 1000 = 2000`
- `bpct = floor(50 / 2000 * 100) = 2%`

Update the assertion from:
```python
self.assertIn("🌾 採集場 Lv1 (100%)", desc)
```
To:
```python
self.assertIn("🌾 採集場 Lv1 (2%)", desc)
```

## Out of Scope

- No change to `building_manager.add_xp` clamping logic.
- No change to `build_main_embed` (it delegates building display through `build_village_embed`).
- No change to the building-target dropdown in the `/idlevillage` view (already shows
  `XP: {bxp}/{next_req}` without a cap override).
- No changes to level-cap calculation (`stages_cleared // 5 + 1`).

## Affected Files

- `src/cogs/ui_renderer.py`
- `tests/test_discord_commands.py`

## Verification

- A Lv1 building with `xp_progress=1000` and `BUILDING_XP_PER_LEVEL=1000` shows 50%,
  not 100%, even when `stages_cleared < 5` (level_cap=1).
- A Lv0 building with `xp_progress=0` shows 0%.
- A building with `xp_progress=next_req` (e.g., 2000 XP at Lv1) shows 100%.
- `test_building_rows_are_plain_percentage_only` passes with the updated assertion.
- No regression in other build_village_embed or build_main_embed tests.

## Dependencies

2026.05.02.00 (UI text and layout refinements) modifies the same building-list rendering
block. This fix must be applied on top of or together with that changelog.
