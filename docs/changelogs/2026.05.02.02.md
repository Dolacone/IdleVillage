# 2026.05.02.02 - Enrich dropdown descriptions for action and gear selection

Status: Draft

## Goal

The action selection dropdown and the gear type dropdown in the `/idlevillage` personal
interface currently show only a label with no contextual information. Players have no
in-UI hint about what each action costs or produces, and no quick view of what upgrading
a gear piece will do. This changelog adds `description` text to every option in both
dropdowns so players can make informed choices without consulting external documentation.

## Implementation Gap

- `build_main_components` produces action options with no `description` field.
- `build_gear_components` produces gear options with no `description` field.
- `build_gear_components` receives no gear-level data and therefore cannot render
  a per-type level-transition hint.
- `_render_gear` in `actions.py` does not fetch all gear levels, only the upgrade_info
  for the currently selected type.

## Scope

### 1. Action dropdown: static cost/output descriptions

File: `src/cogs/ui_renderer.py`, inside `build_main_components`.

Add a static mapping `ACTION_DESCRIPTIONS` at module level (alongside `ACTION_LABELS`):

```
gathering  →  産出 🌾食物 + 🪵木頭
building   →  消耗 🪵木頭 | 産出 建築XP
combat     →  消耗 🪵木頭 | 産出 🧠知識
research   →  消耗 🧠知識 | 産出 研究所XP
```

All actions also consume food (FOOD_COST) as a baseline cost; this is omitted from the
description because it is universal and would add noise. Only the secondary cost that
distinguishes the action from gathering is shown (`wood` for building/combat, `knowledge`
for research).

Pass this as the `description` kwarg to each `disnake.SelectOption` in `action_options`:

```python
action_options = [
    disnake.SelectOption(
        label=f"{ACTION_EMOJIS[a]} {ACTION_LABELS[a]}",
        value=a,
        description=ACTION_DESCRIPTIONS[a],
        default=(pending_action == a),
    )
    for a in ("gathering", "building", "combat", "research")
]
```

No changes to other parts of `build_main_components`.

### 2. Gear type dropdown: dynamic level-transition descriptions

File: `src/cogs/ui_renderer.py`, function `build_gear_components`.

Current signature:
```python
def build_gear_components(gear_type: str, can_attempt: bool) -> list:
```

New signature:
```python
def build_gear_components(
    gear_type: str,
    can_attempt: bool,
    player_gear: dict,
    gear_cap: int,
) -> list:
```

`player_gear` is a flat dict `{action_type: current_gear_level}` e.g.
`{"gathering": 1, "building": 0, "combat": 2, "research": 1}`.

For each gear option, compute the description:

```
if player_gear[g] >= gear_cap:
    description = f"已達等級上限 Lv{gear_cap}"
else:
    bonus_pct = floor(GEAR_BONUS_PER_LEVEL × 100)
    current = player_gear[g]
    current_total = current * bonus_pct
    next_total = (current + 1) * bonus_pct
    description = f"Lv{current} → Lv{current+1}: {ACTION_LABELS[g]}産出 +{current_total}% → +{next_total}%"
```

`GEAR_BONUS_PER_LEVEL` is read with `get_env_float("GEAR_BONUS_PER_LEVEL")` (already
imported by the renderer for other purposes).

Pass this as the `description` kwarg to each `disnake.SelectOption` in `gear_options`.

### 3. Expose player gear levels from `_render_gear` in `actions.py`

File: `src/cogs/actions.py`, function `_render_gear`.

Currently `_render_gear` opens one DB connection to call
`gear_manager.get_upgrade_info(db, user_id, gear_type, now)`.

Extend that same connection to also read all four gear levels from `players`:

```sql
SELECT gear_gathering, gear_building, gear_combat, gear_research
FROM players WHERE user_id=?
```

Build `player_gear = {"gathering": ..., "building": ..., "combat": ..., "research": ...}`
with `0` as the fallback if the row is absent (new player).

Pass `player_gear` and `upgrade_info["gear_cap"]` to `build_gear_components`.

No change to `_render_gear`'s return behaviour or error handling.

## Out of Scope

- No change to action cost values or formulas.

- No change to the embed content of either the main view or the gear upgrade view.
- No change to gear upgrade logic or `gear_manager`.
- No change to the admin interface.

## Verification

- Action dropdown: each option shows a non-empty description matching the table above.
- Gear dropdown at cap: description reads `已達等級上限 Lv{n}`.
- Gear dropdown below cap: description reads `Lv{n} → Lv{n+1}: {type}産出 +{n×pct}% → +{(n+1)×pct}%`
  where `pct = floor(GEAR_BONUS_PER_LEVEL × 100)` and values represent the cumulative bonus at current and next level.
- `build_gear_components` test (if present): update call site to pass `player_gear` and
  `gear_cap`.
- No regression in existing `test_discord_commands.py` tests for `build_main_components`
  or `build_gear_components`.
