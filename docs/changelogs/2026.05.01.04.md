# 2026.05.01.04 - Watcher and public notifications

Status: In-Progress

## Goal

Implement v2 watcher settlement and public notification behavior from `docs/engine/cycle-engine.md` and `docs/discord/notification.md`.

## Implementation Gap

- `src/core/engine.py` currently has a v1 watcher interval and v1 announcement behavior.
- v2 notification ordering and de-duplication are not implemented.
- Stage overtime notification state must live in `stage_state.overtime_notified`.
- Gear success and failure must both send public channel messages.

## Scope

- Implement watcher heartbeat using `WATCHER_HEARTBEAT_SECONDS`.
- Watch due players by `players.completion_time`.
- Settle due complete cycles up to `MAX_CYCLES_PER_SETTLEMENT` per trigger.
- Update the public dashboard after watcher-triggered settlements.
- Send public notifications in documented order:
  - stage clear.
  - upgrade-stage cap context.
  - building upgrades.
  - gear upgrade result when triggered by user action.
  - dashboard update.
- Send stage overtime warning once per stage and set `overtime_notified` to true.
- Reset `overtime_notified` when stage changes.
- Do not persist notification logs.

## Out of Scope

- No new gameplay formulas.
- No command/UI redesign beyond calling existing v2 handlers.
- No multi-guild watcher behavior.

## Files

- `src/core/engine.py`
- `src/cogs/events.py` or watcher startup location
- `src/cogs/general.py` or notification helper module
- focused tests under `tests/`

## Verification

- Watcher interval uses `WATCHER_HEARTBEAT_SECONDS`.
- Due players are settled without waiting for user interaction.
- A stage clear sends a notification for every stage.
- Stage clear overflow does not carry to the next stage.
- Stage overtime warning sends once and does not repeat until stage change.
- Building multi-level upgrades send one message per level.
- Gear success and failure both send public messages with display name, not user id.
- No notification log table or durable log write is introduced.

## Dependencies

Requires `2026.05.01.00`, `2026.05.01.01`, `2026.05.01.02`, and `2026.05.01.03`.
