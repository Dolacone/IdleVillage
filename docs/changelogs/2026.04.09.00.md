# Change Plan & Changelog: 2026.04.09.00

- Status: Done

## 1. Objectives
- Improve the Village Announcement dashboard for better readability and information density.
- Implement localized Last Update timestamp using Discord's relative time format.
- Refine building XP display to show progress relative to the current level rather than lifetime total.
- Aggregate active villager statuses to provide a concise overview of village activities.
- Add a restart-safe migration path so production databases are normalized into the singleton resource-node model.
- Align /idlevillage building XP display with the announcement board.
- Reduce announcement channel noise by disabling discovery messages for exploration results.

## 2. Proposed Changes

### [x] Localized Announcement Header
- Logic: Move the Last Update notice to the very top of the message.
- Format: Use Discord's Unix timestamp with the :R flag (e.g., (Last Update: <t:1234567890:R>)).
- Implementation: Update Engine.render_announcement in src/core/engine.py to calculate and prepend the Unix timestamp.

### [x] Refined Resource Display
- Change: Move the storage capacity (Cap) from the individual resource line to the section header.
- Format: Village Resources (Cap: {max}) followed by 🍎 {food} | 🪵 {wood} | 🪨 {stone}.
- Implementation: Update Engine.render_announcement in src/core/engine.py.

### [x] Relative Building XP Display
- Logic: Subtract the total XP required for the current level from the total accumulated XP to show the Level Progress.
- Formula: Current_Level_XP = Total_XP - Sum_of_XP_Required_for_All_Previous_Levels.
- Format: Building Name: Lv.{lv} [XP: {curr_level_xp} / {next_level_required}].
- Implementation: Update Engine._building_progress_line in src/core/engine.py.

### [x] Aggregated Villager Status
- Logic: Replace the individual list of 20 players with a statistical summary of active tasks.
- Aggregation: Count players grouped by their status and target_id.
- Display Mapping:
  - idle -> Idle
  - exploring -> Exploring
  - gathering -> Gathering {Resource_Type}
  - building -> Building {Building_Name}
- Sorting Rules:
  1. Count (Descending).
  2. Action Name (Ascending, A-Z) as a tie-breaker.
- Implementation: Update the database query and rendering logic in Engine.render_announcement.

### [x] Persistent Singleton Resource Nodes
- Logic: Refactor resource discovery to maintain only one node per type (Food, Wood, Stone) per village.
- Discovery Probability: P(Found) = (PER + KNO) / 4 * 1%.
- Stock Accumulation: New findings add to existing stock, capped at 8,000 per node.
- Weighted Quality Average: New_Quality = floor((Old_Quality * Old_Stock + Found_Quality * Found_Stock) / (Old_Stock + Found_Stock)).
- Expiry Removal: Resource nodes no longer expire but can become Out of Stock.
- Depletion Behavior: When a node hits 0 during settlement, players get partial yield (up to remaining stock), transition to idle status, and receive a notification.
- Implementation: Update Engine.settle_village in src/core/engine.py and the database schema if necessary.

### [x] Restart-safe Resource Node Migration
- Logic: On startup, normalize existing resource nodes into the latest singleton model for this version only.
- Scope:
  - Clear legacy expiry timestamps.
  - Clamp stock into the valid 0..8,000 range.
  - Merge duplicate nodes by (village_id, type).
  - Recalculate merged quality with the documented weighted average and floor rule.
  - Retarget active gathering players from deleted duplicate node ids to the surviving node id.
- Idempotency: The migration must be safe to execute repeatedly on every bot restart without corrupting data or reintroducing duplicates.

### [x] /idlevillage Building XP Alignment
- Logic: The Village Buildings section in /idlevillage must use the same current-level XP progress format as the public announcement.
- Format: Building Name: Lv.{lv} [XP: {curr_level_xp} / {next_level_required}].
- Implementation: Reuse the shared building progress formatter instead of maintaining a second lifetime-XP format in the player command embed.

### [x] Silent Discovery Updates
- Logic: Exploring discoveries still update node stock and quality in the database, but no longer send announcement-channel messages.
- Reason: Discovery frequency is high enough that public discovery notifications create excessive channel noise.
- Documentation Sync: Remove discovery announcement requirements from the announcement and resources interface docs.
