# 2026.05.04.01 - Fix stage bonus to count upgrade clears

Status: Done

## Goal

The current efficiency formula treats `STAGE_BONUS_PER_CLEAR` as a bonus for
every cleared stage. This is incorrect. Stage bonus should mean the number of
upgrade stages cleared, not the total number of stages cleared.

Observed example:

```
stages_cleared = 19
facility level = 4
player gear level = 4

current stage bonus    = 19 * 0.01 = 19%
gear bonus             = 4 * 0.03 = 12%
facility bonus         = 4 * 0.01 = 4%
current total display  = +35%
```

Expected example:

```
stages_cleared = 19
upgrade clears = floor(19 / 5) = 3
facility level = 4
player gear level = 4

stage bonus            = 3 * 0.01 = 3%
gear bonus             = 4 * 0.03 = 12%
facility bonus         = 4 * 0.01 = 4%
expected total display = +19%
```

## Design

Stage bonus must be calculated from completed upgrade stages:

```
upgrade_clears = floor(stages_cleared / 5)
stage_bonus = upgrade_clears * STAGE_BONUS_PER_CLEAR
```

This rule must apply consistently to both production output and UI efficiency
display. `stages_cleared` remains the canonical database field for total
cleared stages. No new database field is required.

The fifth stage in each cycle is the upgrade stage, so the bonus increases only
after clear counts 5, 10, 15, 20, and so on.

## Scope

### 1. Update the formula documentation

File: `docs/engine/formula.md`.

Change the stage bonus definition from total cleared stages to completed
upgrade stages:

```
stage_bonus = floor(stages_cleared / 5) * STAGE_BONUS_PER_CLEAR
```

Clarify that `STAGE_BONUS_PER_CLEAR` means bonus per cleared upgrade stage.

### 2. Update production output calculation

File: `src/core/formula.py`, function `compute_output`.

Replace direct use of total `stages_cleared` in the bonus expression with
`stages_cleared // 5`.

No change to gear bonus, facility bonus, action mappings, or output flooring.

### 3. Update personal efficiency display

File: `src/cogs/ui_renderer.py`, function `build_main_embed`.

Use the same `stages_cleared // 5` stage bonus input when rendering the
efficiency line. The UI must not display a different percentage from the value
used by `compute_output`.

### 4. Update UI documentation

File: `docs/discord/ui-renderer.md`.

Update the efficiency line notes to reference the corrected formula. The
renderer doc should not redefine stage bonus differently from
`docs/engine/formula.md`.

### 5. Update tests

Update existing tests that assume stage bonus equals total cleared stages.

Required cases:

- `stages_cleared = 4` produces zero stage bonus.
- `stages_cleared = 5` produces one stage bonus.
- `stages_cleared = 19` produces three stage bonuses.
- Main embed efficiency display matches `compute_output` semantics.

## Out of Scope

- No change to stage progression, stage targets, or stage clear state.
- No change to building level caps.
- No change to `STAGE_BONUS_PER_CLEAR` environment variable name in this
  changelog.
- No change to database schema.
- No migration is required.

- With `stages_cleared = 19`, gear level 4, facility level 4, and the current
  environment values, the displayed bonus is `+19%`, not `+35%`.
- `compute_output` and the personal efficiency display use the same stage bonus
  input.
- Existing gear and facility bonuses are unchanged.

- Stage clear count and upgrade stage progression behavior are unchanged.
