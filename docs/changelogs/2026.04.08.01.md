## 2. Proposed Changes

### [x] Identity Mapping: Discord IDs as Primary Gameplay Keys
- Completed Change: Use Discord Guild ID directly as the village primary key, and identify players by `(discord_id, village_id)` so the same user can safely exist in multiple guilds.
- Reason: The project now treats Discord IDs as stable numeric identifiers, and the composite player identity prevents cross-guild row collisions.
- Verification: Added regression coverage for the same Discord user existing in multiple villages without state collision.

### [x] Bug 1: Multi-Cycle Catch-up Failure (Settlement Leap)
- Bugged Scenario: A player is away for multiple cycles. Upon returning, they only receive 1 cycle's worth of resources and XP.
- Reproduce Steps: Set ACTION_CYCLE_MINUTES=1. Start an action. Close bot/wait 5 mins. Reopen and check resources. Only 1 cycle added.
- Root Cause: settle_player only settles the active cycle and resets completion_time to Now + Cycle, jumping over missed cycles.
- Expected Behavior: Implement iterative while loop in settle_player to backfill all missed cycles until caught up or resources empty.

### [x] Bug 2: Background Watcher Loop Death
- Bugged Scenario: Village building XP never decays and players never auto-settle.
- Reproduce Steps: Check village XP. Wait 10 mins without commands. Check XP again. No change.
- Root Cause: @tasks.loop decorator is incompatible with @staticmethod.
- Resolution Note: Current build verification shows `Engine.start_watcher_loop` is a live `disnake.ext.tasks.Loop`, starts correctly from `main.py`, and the failure no longer reproduces.

### [x] Bug 4: XP Decay Integer Truncation
- Bugged Scenario: Buildings never lose XP under high-frequency checks.
- Reproduce Steps: Set Watcher to 60s. Monitor XP. Decay remains 0.
- Root Cause: int(hours * rate) truncates to 0 for small time deltas.
- Expected Behavior: Use cycle-unit scaling: decay = (Delta_Minutes / ACTION_CYCLE_MINUTES) * (10 + active_count).

### [x] Bug 6: Missing-Status Leak
- Bugged Scenario: Inactive players are permanently counted as active, inflating village decay.
- Reproduce Steps: Inactive account for 8+ days still contributes to active_count in village decay logs.
- Root Cause: missing status logic is trapped inside individual player settlement which requires interaction.
- Expected Behavior: Implement a decoupled global scan task to mark players missing only when both guild-message time and `/idlevillage` command time are older than 7 days.

### [x] Bug 3 & 5: Logic Propagation Fixes
- Bug 3 (Food Cost): Ensure transactional commit of resource deduction before updating action deadlines.
- Bug 5 (Stats Lag): Recalculate stats inside the backfilling loop for accurate per-cycle yield.

## 5. Changelog
- Recorded the Discord identity-key refactor so villages now use Guild ID directly and players are isolated by `(discord_id, village_id)`.
- Reworked player settlement to backfill overdue cycles iteratively, applying one completed cycle at a time before starting the next prepaid lease.
- Recalculated player stats before each backfilled cycle so cycle-by-cycle outputs stay consistent with the 150-cycle stat window.
- Moved missing-player detection into a decoupled inactivity scan used by watcher and village decay, preventing abandoned idle players from inflating active-count decay.
- Switched village decay to cycle-based scaling so short heartbeat intervals still apply XP loss correctly.
- Added regression coverage for overdue backfill, per-cycle stat propagation, short-heartbeat decay, and watcher-driven missing transitions.
- Refined the inactivity baseline so `missing` now depends only on guild-message time plus `/idlevillage` command time, and no longer treats in-game actions as activity.
