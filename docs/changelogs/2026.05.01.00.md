# 2026.05.01.00 - Foundation: v2 config and fresh DB schema

Status: Done

## Goal

Replace the v1 database and configuration foundation with the v2 fresh-start foundation described in `docs/db-schema.md` and `docs/engine/formula.md`.

## Implementation Gap

- `src/database/schema.py` still creates and migrates v1 tables such as villages, buffs, player_stats, resource_nodes, monsters, tokens, and player_buffs.
- `src/core/config.py` only exposes v1-era admin and action-cycle helpers.
- Startup does not validate the full v2 environment contract.
- Required v2 environment keys are not centralized or validated.
- Current tests still assert v1 schema and mechanics.

## Scope

- Replace schema initialization with v2-only tables:
  - `village_state`
  - `stage_state`
  - `village_resources`
  - `buildings`
  - `players`
  - `guild_installations`
- Seed initial singleton village/stage rows and required resource/building rows.
- Treat every key listed in `.env.example` as required at runtime.
- Fail startup when any listed environment key is missing.
- Load all v2 environment keys through centralized typed config helpers.
- A shared typed getter pattern is acceptable, but raw `os.getenv` calls for gameplay/config keys must not be scattered across modules.
- Seed `stage_state.current_stage_target` from the stage target formula using `STAGE_BASE_TARGET`.
- Detect existing v1 database tables and fail startup with a clear message.
- Keep async SQLite access through `aiosqlite`.
- Keep the Discord bot startup shell intact.

## Out of Scope

- No gameplay settlement implementation.
- No Discord UI rewrite.
- No v1 migration path.
- No data preservation from existing v1 databases.
- No automatic drop, truncate, rename, or destructive cleanup of existing v1 database files. The operator manually deletes or replaces the DB.

## Files

- `src/core/config.py`
- `src/database/schema.py`
- `src/main.py`
- `.env.example`
- focused tests under `tests/`

## Verification

- Schema initialization creates only v2 tables.
- Required seed rows exist after initialization.
- Missing required environment keys print the missing key names and stop startup.
- Every key listed in `.env.example` is validated as present.
- Config helpers return typed values and do not hardcode gameplay values.
- Existing v1 database tables cause startup failure instead of automatic cleanup.
- Initial `stage_state.current_stage_target` equals the stage-manager initial target derived from `STAGE_BASE_TARGET`.
- No test relies on v1 tables.

## Dependencies

None. This must be implemented before every later section.
