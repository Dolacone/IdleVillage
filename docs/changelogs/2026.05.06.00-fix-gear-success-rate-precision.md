# 2026.05.06.00 - Fix gear success-rate precision

Status: Done

## Goal

Gear upgrade success-rate calculation and display must follow the documented
decimal formula exactly enough for configured integer-percent rates to remain
integer-percent rates.

Observed case:

```
gear type = gathering
current level = 6
target level = 7
pity_count = 0
GEAR_RATE_LOSS_PER_LEVEL = 0.10

documented rate = 100% - 6 * 10% = 40%
current float value = 0.3999999999999999
current UI display = 39%
```

Expected case:

```
computed final_rate = 0.40
UI display = жҲҗеҠҹзҺҮпјҡ40%пјҲ+0Г—5% дҝқеә•пјү= 40%
```

## Implementation Gap

`docs/managers/gear-manager.md` defines the success-rate formula in decimal
terms, but the current implementation uses raw binary floating-point arithmetic.
For Lv6 with `GEAR_RATE_LOSS_PER_LEVEL=0.10`, Python evaluates
`1.0 - 6 * 0.10` as `0.3999999999999999`.

`src/cogs/ui_renderer.py` then floors `rate * 100`, so the UI displays 39 even
though the documented rate is 40.

The same underflow also affects the actual random success check, which compares
against `0.3999999999999999` instead of the intended `0.40`.

## Design

Normalize gear upgrade success-rate values at the gear-manager boundary so the
calculated value matches the documented decimal intent.

The normalization must apply to:

- `base_rate`
- `final_rate`
- the rate returned by `get_upgrade_info`
- the rate used by `attempt_upgrade`

The UI must format the percentage from the normalized rate, so the displayed
percentage cannot drift below the calculated percentage due to floating-point
underflow.

The change should preserve the existing formula:

```
base_rate = max(GEAR_MIN_SUCCESS_RATE, 100% - current_level * GEAR_RATE_LOSS_PER_LEVEL)
final_rate = min(100%, base_rate + pity_count * GEAR_PITY_BONUS)
```

## Scope

### 1. Update gear success-rate calculation

File: `src/managers/gear_manager.py`.

Adjust `_compute_rate` so decimal config values such as `0.10` produce the
documented success rate. Lv6 with no pity must return exactly `0.40`.

Do not change upgrade preconditions, AP spending, material spending, pity
increment behavior, cap behavior, or random roll semantics beyond using the
correct normalized rate.

### 2. Update gear success-rate display

File: `src/cogs/ui_renderer.py`.

Ensure the gear upgrade embed formats normalized rate values. Lv6 with no pity
must display:

```
жҲҗеҠҹзҺҮпјҡ40%пјҲ+0Г—5% дҝқеә•пјү= 40%
```

Do not change unrelated dashboard efficiency rendering.

### 3. Update tests

Files:

- `tests/test_gear_manager.py`
- `tests/test_discord_commands.py`

Add focused coverage for the Lv6 precision boundary:

- `_compute_rate(6, 0) == 0.40`
- `build_gear_embed` for Lv6 and pity 0 includes
  `жҲҗеҠҹзҺҮпјҡ40%пјҲ+0Г—5% дҝқеә•пјү= 40%`

Existing min-rate, max-rate, pity, precondition, and display tests must remain
valid.
