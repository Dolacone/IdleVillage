# 2026.05.02.03 - Fix building XP cap: allow accumulation up to next_req when capped

Status: Draft

## Goal

When a building is at level cap, XP should accumulate up to the full upgrade threshold
(`(level + 1) × BUILDING_XP_PER_LEVEL`), holding at 100% until the cap is lifted and the
building immediately upgrades. Currently, the clamp is applied at `level × BUILDING_XP_PER_LEVEL`,
which is one step short of the threshold, so a capped building can only reach 50% display at
Lv1 and diminishing fractions at higher levels.

## Root Cause

In `src/managers/building_manager.py`, `add_xp` clamps incoming XP at `level * xp_per`
instead of `(level + 1) * xp_per`. The display formula in `ui-renderer.py` is correct
(`floor(xp_progress / next_req * 100)`) and requires no change — 100% is reached naturally
once `xp_progress` is allowed to reach `next_req`.

## Documentation Notes

`docs/managers/building-manager.md` and `docs/discord/ui-renderer.md` have already been
updated as part of this plan (Phase 1 SSOT sync). The cap rule now reads
`(level + 1) × BUILDING_XP_PER_LEVEL` and the special-case display override has been
removed. The builder should treat the current docs as authoritative.

## Scope

### 1. Fix XP clamp in building_manager

File: `src/managers/building_manager.py`, function `add_xp`.

Two clamp sites must be updated. Both must use `(level + 1) * xp_per` instead of
`level * xp_per`.

Site A — early-return branch (building already at cap on entry):

```python
# current
clamped = min(xp_progress + xp, level * xp_per)

# corrected
clamped = min(xp_progress + xp, (level + 1) * xp_per)
```

Site B — post-upgrade-loop clamp (building may have just reached cap during this call):

```python
# current
xp_progress = min(xp_progress, level * xp_per)

# corrected
xp_progress = min(xp_progress, (level + 1) * xp_per)
```

No other changes to `building_manager.py`.

### 2. Fix tests in test_engine_settlement.py

File: `tests/test_engine_settlement.py`, `test_building_capped_at_level_cap`.

The test should express the correct design: a building at level cap accumulates XP up to
exactly `(level + 1) × xp_per` (the upgrade threshold), then stops. It does not upgrade.

Pre-seed `xp_progress = 2 * xp_per - 1` so that incoming settlement XP (`base_output = 20`)
would push it past the cap. Replace the `UPDATE buildings` seed and the final assertions:

```python
# seed xp_progress just below the correct cap so settlement XP pushes it over
await db.execute(
    "UPDATE buildings SET level=1, xp_progress=? WHERE building_type='gathering_field'",
    (2 * xp_per - 1,),
)

# ...after settlement...

# Level must still be 1 (building is capped, not eligible to upgrade)
self.assertEqual(bld_after["level"], 1)
# xp_progress must be clamped to exactly (level+1)*xp_per = 2*xp_per
self.assertEqual(bld_after["xp_progress"], 2 * xp_per)
```

### 3. Fix tests in test_discord_commands.py

File: `tests/test_discord_commands.py`, `TestRendererVillageEmbed`.

Remove the incorrectly-premised test `test_capped_building_shows_100_percent` (added during
bug analysis, assumes a display override at any xp value).

Replace it with a test that uses `xp_progress = next_req = 2 * BUILDING_XP_PER_LEVEL` on a
capped Lv1 building and asserts 100% via the natural formula:

```python
def test_capped_building_at_full_xp_shows_100_percent(self):
    from cogs.ui_renderer import build_village_embed

    xp_per = int(ALL_TEST_ENV["BUILDING_XP_PER_LEVEL"])
    # stages_cleared=3 → level_cap=1; gathering_field Lv1 with full next_req XP
    buildings = {
        "gathering_field": {"level": 1, "xp_progress": 2 * xp_per},
    }
    embed = build_village_embed(self._make_stage_data(), {}, buildings, [])
    self.assertIn("🌾 採集場 Lv1 (100%)", embed.description)
```

No change to `test_capped_building_row_shows_actual_xp_percentage` — its assertions remain
valid (50% for xp_progress=1000 on a 2000-threshold bar is correct intermediate progress).

### 4. No change to ui_renderer.py

The display formula `pct = floor(xp_progress / next_req * 100)` is correct as-is. The 100%
is achieved naturally once the clamp is fixed. Do not add a level-cap override to the
renderer.

## Out of Scope

- No change to upgrade logic, level-cap calculation, or cap-lift behaviour.
- No change to `check_all_upgrades` — its logic is correct.
- No change to any other test files.
- No change to gear or facility logic.

## Verification

- Capped Lv1 building with `xp_progress = 0` displays 0%.
- Capped Lv1 building with `xp_progress = xp_per` displays 50%.
- Capped Lv1 building with `xp_progress = 2 * xp_per` displays 100%.
- After cap lifts to Lv2, building with full XP immediately upgrades to Lv2.
- `test_capped_building_at_full_xp_shows_100_percent` passes.
- `test_building_capped_at_level_cap` passes with corrected assertion.
- No regression in other settlement or renderer tests.
