# Change Plan & Changelog: 2026.04.14.00

- Status: Done

## 1. Objectives
- Redesign `player_actions_log` so it stores per-cycle stat gains instead of action names.
- Align stat progression with completed Action Cycles only, so partial settlement slices do not consume the 150-cycle stat window.
- Limit `player_actions_log` to the newest 150 records per player and calculate cached player stats by summing those retained records.

## 2. Proposed Changes

### Database Updates (`src/database/schema.py`)
- [x] Redefine `player_actions_log` from action-history rows to stat-history rows.
- [x] Replace `action_type`, `start_time`, and `end_time` as the stat-calculation source with per-cycle stat delta columns:
  - `strength_delta`
  - `agility_delta`
  - `perception_delta`
  - `knowledge_delta`
  - `endurance_delta`
- [x] Keep a cycle ordering field so the newest 150 records per player can be retained deterministically.
- [x] Enforce the retention rule during writes: after inserting a completed-cycle stat row, delete older rows for that `(player_discord_id, village_id)` pair so only the newest 150 records remain.

### Engine Logic (`src/core/engine.py`)
- [x] Write to `player_actions_log` only when a full Action Cycle is completed (`now >= completion_time` and the settled slice covers a full cycle).
- [x] Stop writing partial settlement slices from UI refresh, interruptions, or other sub-cycle settlements into `player_actions_log`.
- [x] Convert each completed cycle into explicit stat deltas instead of storing an action label:
  - `idle`: `perception_delta = 1`, `knowledge_delta = 1`
  - `gathering_food`: `perception_delta = 1`, `knowledge_delta = 1`
  - `gathering_wood`: `strength_delta = 1`, `endurance_delta = 1`
  - `gathering_stone`: `strength_delta = 1`, `endurance_delta = 1`
  - `exploring`: `agility_delta = 1`, `perception_delta = 1`
  - `building`: `knowledge_delta = 1`, `endurance_delta = 1`
  - `attack`: `strength_delta = 1`, `agility_delta = 1`
- [x] Recalculate `player_stats` as `STATS_BASE_VALUE + SUM(delta columns)` across the retained `player_actions_log` rows for that player.
- [x] Preserve the existing efficiency formula `(StatA + StatB) / 2 / 100`; this change only affects how stat history is recorded and replayed.

### Documentation Updates (`docs/modules/player_stats.md`, `docs/core/engine.md`, `docs/terminology.md`)
- [x] Update the SSOT wording so `player_actions_log` is defined as a 150-cycle stat-history table, not a generic action-history table.
- [x] Clarify that only completed cycles contribute to stat growth.
- [x] Clarify that the table itself is capped at 150 records per player, so stat recalculation sums all retained rows.

### Automated Test Updates (`tests/test_player_stats_module.py`, `tests/test_village_module.py`)
- [x] Verify a completed cycle inserts exactly one stat-delta row.
- [x] Verify partial settlements do not insert stat-delta rows.
- [x] Verify the 151st completed cycle evicts the oldest retained row and keeps only 150 rows for that player.
- [x] Verify `player_stats` equals base stats plus the sum of retained stat-delta rows.
