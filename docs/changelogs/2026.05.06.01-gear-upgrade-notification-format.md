# 2026.05.06.01 - Gear upgrade notification format

Status: Done

## Goal

Gear upgrade Public notifications must show all information needed to understand
the attempted upgrade result in one line:

- current level
- target level
- success or failure
- total failure count

The official user-facing name for gear is 工具. Full gear names are:

- 採集工具
- 建設工具
- 狩獵工具
- 研究工具

The personal dashboard uses the line label `🏅 工具`.

Required format:

```
玩家名稱 的 採集工具 升級成功 :tada: Lv1 -> Lv2（總失敗次數：0）
玩家名稱 的 採集工具 升級失敗 :boom: Lv1 -> Lv2（總失敗次數：3）
```

## Implementation Gap

Current notification formatting is split by result:

- success only shows `new_level`
- failure shows `current_level` and `pity_count`

The success path does not include `current_level`, `target_level`, or the
failure count before pity reset. The failure path does not explicitly show the
target level.

`gear_manager.attempt_upgrade` already returns `current_level`, `new_level`, and
`pity_after`, but it does not return the pre-roll pity count needed to display
the success message's total failure count before reset.

## Design

Use one shared display contract for both gear success and gear failure events.

Event payload fields:

- `user_display_name`
- `gear_type`
- `current_level`
- `target_level`
- `failure_count`

Field meanings:

- `current_level`: gear level before this attempt.
- `target_level`: attempted upgrade level, always `current_level + 1`.
- `failure_count`: total accumulated failures for this gear.

For success, `failure_count` means the number of failures accumulated before
the successful attempt reset pity to 0.

For failure, `failure_count` means the number of failures accumulated after this
failed attempt, so it includes the current failure.

Notification templates:

```
{user_display_name} 的 {gear_name} 升級成功 :tada: Lv{current_level} -> Lv{target_level}（總失敗次數：{failure_count}）
{user_display_name} 的 {gear_name} 升級失敗 :boom: Lv{current_level} -> Lv{target_level}（總失敗次數：{failure_count}）
```

## Scope

### 0. Update official user-facing tool naming

Files:

- `docs/managers/gear-manager.md`
- `docs/managers/player-manager.md`
- `docs/engine/formula.md`
- `docs/discord/ui-renderer.md`
- `docs/discord/notification.md`
- `src/cogs/ui_renderer.py`
- relevant tests

Use these user-facing names everywhere UI or public messages display a specific
gear/tool name:

- gathering: 採集工具
- building: 建設工具
- combat: 狩獵工具
- research: 研究工具

Use `工具` for the personal dashboard category label.

### 1. Update gear upgrade result data

File: `src/managers/gear_manager.py`.

Return the pre-roll pity count from `attempt_upgrade`, for example as
`pity_before`, so the caller can build a correct success notification.

Do not change AP spending, material spending, success-rate calculation, roll
logic, gear level changes, pity increment, or pity reset behavior.

### 2. Update gear event creation

File: `src/cogs/actions.py`.

For both success and failure events, include:

- `current_level`
- `target_level`
- `failure_count`

Use `pity_before` for success `failure_count`.
Use `pity_after` for failure `failure_count`.

### 3. Update notification formatting

File: `src/core/notification.py`.

Format `gear_success` and `gear_fail` using the exact templates in this plan.

### 4. Update tests

Files:

- `tests/test_discord_notifications.py`
- `tests/test_discord_commands.py`

Add or update focused assertions covering:

- success message includes `升級成功 :tada:`
- failure message includes `升級失敗 :boom:`
- both messages include `Lv{current_level} -> Lv{target_level}`
- both messages include `總失敗次數：{failure_count}`
- success event creation uses pre-reset failure count
- failure event creation uses post-failure count
