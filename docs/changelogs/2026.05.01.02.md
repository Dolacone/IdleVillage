# 2026.05.01.02 - Managers: stage, building, player, gear, resources

Status: Done

## Goal

Implement the v2 manager layer from `docs/managers/` on top of the v2 engine and schema.

## Implementation Gap

- v1 code models resources, buildings, player stats, tokens, buffs, and monsters instead of v2 stage/building/player/gear/material systems.
- Building XP is currently cumulative/exponential in v1, while v2 uses current-level progress with linear requirements.
- Player AP and gear material systems are absent in v2 form.

## Scope

- Implement resource manager for food, wood, and knowledge.
- Implement player manager:
  - single-guild player identity.
  - `ap_full_time` AP model.
  - new players start at 0 AP.
  - material balances.
- Implement stage manager:
  - five-stage loop.
  - linear round difficulty.
  - upgrade stage target multiplier.
  - stage progress overflow discarded on clear.
  - overtime progress multiplier and `overtime_notified` flag.
- Implement building manager:
  - four building types.
  - level starts at 0.
  - current-level `xp_progress`.
  - capped progress at level cap.
  - cap increase triggers immediate upgrade checks.
- Implement gear manager:
  - AP and material costs.
  - success rate formula.
  - pity counter.
  - success/failure state updates.

## Out of Scope

- No Discord UI rendering.
- No watcher loop.
- No public message sending.

## Files

- new or existing manager modules under `src/`
- `src/core/engine.py` integration points
- focused tests under `tests/`

## Verification

- Stage clear increments `stages_cleared`, advances stage type, resets progress to 0, and resets `overtime_notified` to false.
- Stage overtime only affects stage progress income.
- Building level cap prevents level increase and clamps progress to the current requirement.
- Building cap increase immediately upgrades eligible buildings and preserves valid surplus.
- Gear success resets pity and increases level.
- Gear failure keeps level and increments pity.
- AP calculations derived from `ap_full_time` match the documented model.

## Dependencies

Requires `2026.05.01.00` and `2026.05.01.01`.
