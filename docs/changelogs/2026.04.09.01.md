# Change Plan & Changelog: 2026.04.09.01

- Status: Done

## 1. Objectives
- Normalize the database schema by separating resources and buildings (as "buffs") into independent tables for future extensibility.
- Implement a restart-safe, one-time migration to transition from 2026.04.09.00 schema to 2026.04.09.01.
- Implement building (buff) level-down notifications caused by XP decay.
- Perform a comprehensive numerical rebalance to standardize costs, outcomes, and formulas.
- Refactor rendering and calculation logic into shared Engine methods to ensure long-term consistency.

## 2. Proposed Changes

### [x] Schema Normalization & One-time Migration (Extensibility)
- Create village_resources table: (village_id, resource_type, amount) with one row per village/resource pair.
- Create buffs table: (village_id, buff_id, xp) with one row per village/buff pair.
- Backup-first Migration Guardrail:
  - Before any destructive schema step, create a full copy of the current SQLite DB file beside the original DB file using a versioned or timestamped `.bak` suffix.
  - If the backup step fails, abort the migration and keep the legacy schema untouched.
- Restart-safe / Idempotent Migration Logic:
  - Detect whether legacy columns still exist in villages table.
  - Create normalized tables first if they do not already exist.
  - Port food, wood, stone to village_resources.
  - Port food_efficiency_xp, storage_capacity_xp, resource_yield_xp to buffs (IDs 1, 2, 3).
  - If normalized rows already exist for a village/resource or village/buff pair, treat the normalized row as the source of truth and do not duplicate or add the legacy value again.
  - Drop legacy columns from villages table only after backup creation and post-copy verification succeed for all villages.
- Refactor: Update Engine to perform dynamic lookups from village_resources and buffs tables.

### [x] Numerical Rebalance (Core Constants)
- Centralize Constants:
  - BASE_OUTCOME = 50
  - BASE_BUILD_COST = 50
  - BASE_FOOD_COST = 20
  - STATS_BASE_VALUE = 50
  - EXPLORING_BASE_CHANCE = 0.2
- Standardize Outcomes: All gathering and building actions use BASE_OUTCOME (50).
- Standardize Costs: 
  - Building: 50 Wood and 50 Stone per cycle.
  - Food: max(10, 20 - Kitchen_Level).
- Initial Resources: New villages initialize with 1,000 Food, 1,000 Wood, and 1,000 Stone in village_resources.
- Calibrate Exploring:
  - Exploring Efficiency: (AGI + PER) / 2 / 100.
  - Discovery Chance: Exploring_Efficiency * EXPLORING_BASE_CHANCE.
  - Discovery Stock: BASE_OUTCOME * random(20, 40) * Exploring_Efficiency.
  - Discovery Result: Update the singleton resource node for that type by creating it if absent, or replenishing / recomputing quality if it already exists.
- Resource Yield Placement:
  - Resource Yield buff remains a multiplicative modifier on gathering output before storage-cap truncation.

### [x] Building Level-Down Announcements
- Logic: During settle_village (decay), compare levels of items in the buffs table before and after XP deduction.
- Event: If current_level < previous_level, send an announcement message.

### [x] Logic Consolidation (Refactor)
- Formula Unification: Create Engine.calculate_efficiency and Engine.calculate_outcome.
- Rendering Consolidation: Create shared methods for rendering building (buffs) and villager summaries.

## 4. Verification Plan

### 4.1 Backup & Migration Verification
- Scenario: Existing production DB is still on the 2026.04.09.00 schema and migration starts on bot boot.
- Expected:
  - A full DB backup file is created before any schema rewrite or column drop occurs.
  - If backup creation fails, the migration aborts without modifying the live DB schema.
  - If the bot restarts after normalized tables are created but before legacy columns are dropped, re-running the migration does not duplicate rows or inflate values.
  - village_resources has row ('food', 100).
  - village_resources has rows for 'wood' and 'stone' that match legacy values.
  - buffs has rows (1, food_efficiency_xp), (2, storage_capacity_xp), (3, resource_yield_xp).
  - villages table no longer contains food or food_efficiency_xp columns.

### 4.2 Numerical Balance Verification
- Scenario: Player with 100/100 stats performs Building.
- Expected: Cost 50/50, Food cost 18 (Lv2 Kitchen), Gain 50 XP.

### 4.3 Exploring Verification
- Scenario: Player with AGI 100 and PER 100 performs Exploring for one full cycle.
- Expected:
  - Efficiency = 1.0.
  - Discovery chance uses 20% base rate.
  - Successful discovery adds BASE_OUTCOME * random(20, 40) stock to the singleton node for the discovered type.
  - No standalone discovery announcement message is posted.

### 4.4 Level-Down Verification
- Scenario: XP for a buff drops below a level threshold during decay.
- Expected: Level decreases and announcement is sent.

### 4.5 New Village Initialization
- Scenario: A brand-new village is initialized after this version is deployed.
- Expected:
  - village_resources starts with food/wood/stone = 1,000 each.
  - buffs contains IDs 1, 2, 3 initialized to XP 0.
  - villages contains only village metadata columns required by the normalized schema.
